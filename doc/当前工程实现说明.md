# SinglePTZ-FaceTrack 当前工程实现说明

这份文档以当前代码实现为准，目标是让你快速看清这个工程现在到底做了什么、主流程怎么走、检测/PTZ/识别分别是怎么串起来的。

## 1. 这个项目现在在做什么

一句话概括：

这个项目控制一台海康球机在多个预置位之间巡航，在巡航画面里先找人，再把球机拉近到目标区域，在近景里抓取人脸，最后用 ArcFace 和本地 `photo/` 人脸库做识别，并把结果记到 `output/events.jsonl`。

当前主流程不是“远景直接识别人脸”，而是：

1. 预置位巡航。
2. 在远景画面里先检测“人”。
3. 对每个人体框做 PTZ 拉近。
4. 在近景里再检测“脸”并抓拍。
5. 对抓到的人脸做识别。
6. 识别命中后，利用 ReID 避免同一轮扫描里在别的预置位重复识别同一个人。

## 2. 运行入口和启动链路

入口文件：

- `run.sh`
- `src/main.py`

启动时实际做的事情：

1. `run.sh` 设置海康 SDK 的 `LD_LIBRARY_PATH`，激活 `single_ptz_facetrack` conda 环境，然后运行 `python src/main.py`。
2. `src/main.py` 读取 `config/config.yaml`，初始化日志。
3. 初始化 ISAPI 客户端 `HikISAPI`。
4. 初始化海康 SDK 封装 `HikSDK` 和 PTZ 业务封装 `HikPTZ`。
5. 登录球机，并检查设备是否支持 3D 定位能力。
6. 加载模型：
   - `YoloFace(face_wide)`
   - `YoloFace(face_close)`
   - `YoloPerson(person)`
   - `ArcFace`
   - `OSNetReID`
7. 用 `photo/` 构建或加载本地人脸库缓存 `cache/gallery.npz`。
8. 启动 RTSP 拉流线程 `RtspSource`。
9. 创建并运行状态机 `ScanScheduler`。

## 4. 一次完整业务流程

### 4.1 巡航发现阶段

状态机从 `INIT` 进入 `PATROL_GOTO` 和 `PATROL_DWELL`。

当前配置里巡航预置位是：

```yaml
patrol:
  presets: [1, 2, 3, 4]
  dwell: 4.0
  min_confirm_frames: 3
```

具体流程：

1. `PATROL_GOTO`
   - 调用 `ptz.goto_preset(preset)` 去指定预置位。
   - 记录本次 PTZ 命令时间，后面用来判断何时稳定。

2. `PATROL_DWELL`
   - 先等画面稳定。
   - 画面稳定后开始驻留 `dwell` 秒。
   - 在驻留期间，每帧做一次检测。
   - 当前代码优先用 `person_det`，只有 `person_det` 不存在时才退回 `face_wide`。
   - 连续 `min_confirm_frames` 帧检测到目标后，认为这个预置位“有人”，进入扫描模式。

所以当前代码里，“巡航触发扫描”的依据主要是**人形检测**，不是远景人脸检测。

### 4.2 进入扫描模式

一旦某个预置位触发，状态机会把这一轮扫描顺序构造成：

1. 从触发预置位开始。
2. 按配置里的预置位顺序循环一整圈。

例如触发位是 `3`，则扫描顺序会变成 `[3, 4, 1, 2]`。

### 4.3 每个预置位的扫描流程

对当前扫描预置位，状态机会依次执行：

1. `SCAN_GOTO_PRESET`
   - 再次跳转到该预置位。
   - 等待画面稳定。

2. `SCAN_DETECT`
   - 远景里做检测，当前仍然优先用 `person_det`。
   - 如果启用了 `Tracker`，会把检测结果送入 `Tracker.update()`。
   - 跟踪器会给每个目标维护 ReID 特征。
   - 如果某个目标的 ReID 特征和本轮已经识别成功的人相似度超过阈值 `reid.cross_preset_th=0.5`，就跳过，不再重复扫描。

3. `SCAN_PICK`
   - 把本预置位的候选目标排序。
   - 排序规则是：
     - 先按框面积从大到小。
     - 面积相同时按从左到右。
   - 取队首目标，调用 `ptz.zoom_to_bbox()` 拉近。

4. `SCAN_SETTLE`
   - 等待 PTZ 运动结束，等待画面稳定。
   - 再做一次对焦稳定判定。
   - 稳定后进入抓拍阶段。

5. `SCAN_CAPTURE`
   - 在近景画面里用 `face_close` 检测人脸。
   - 收集满足质量要求的人脸 crop。
   - 达到最少样本数或者超时后，进入识别。

6. `SCAN_RECOGNIZE`
   - 对抓到的人脸做 ArcFace embedding。
   - 多张特征求平均后与人脸库匹配。
   - 写入事件日志。

7. `SCAN_ZOOM_OUT`
   - 回到当前扫描预置位。
   - 然后继续扫描本预置位剩余目标。

8. `SCAN_NEXT_PRESET`
   - 当前预置位的目标处理完后，切到下一预置位。
   - 一整轮都扫完后，清空本轮去重表，回到巡航状态。

## 5. 检测逻辑

## 5.1 当前用了哪些检测器

项目里有三类检测输入：

| 场景 | 模型 | 当前是否在主流程中生效 |
| --- | --- | --- |
| 远景发现目标 | `YoloPerson` | 是，主流程优先使用 |
| 远景人脸检测 | `YoloFace(face_wide)` | 只有 `person_det` 不可用时才作为回退 |
| 近景抓脸 | `YoloFace(face_close)` | 是，抓拍阶段核心检测器 |

这点很重要：

当前系统虽然加载了 `face_wide`，但由于 `main.py` 总是初始化 `person_det`，所以巡航和扫描阶段实际上主要靠 **YOLOv8 person** 在工作。

## 5.2 远景检测怎么做

文件：

- `src/detect/yolo_person.py`

逻辑：

1. 先把输入帧做 letterbox 到 `640x640`。
2. ONNX Runtime 推理。
3. 只取 COCO `person` 类。
4. 用 `person_conf` 和 `person_iou` 做阈值过滤和 NMS。
5. 把框映射回原始 RTSP 画面坐标。
6. 如果开启边缘过滤，则丢弃贴边框。

当前配置：

```yaml
detect:
  person_conf: 0.45
  person_iou: 0.5
  edge_reject_enabled: true
  edge_margin: 5
```

也就是说，远景检测阶段会主动过滤离画面边缘 5 像素以内的目标。

## 5.3 近景抓脸怎么做

文件：

- `src/detect/yolo_face.py`
- `src/scheduler/capture_tracker.py`

流程：

1. 在 CAPTURE 阶段，每帧用 `face_close` 做一次人脸检测。
2. 如果检测到多张脸，取“最接近画面中心”的那一张。
3. 如果这张脸中心落在安全区域内，则尝试采样。
4. 采样时优先调用 ISAPI 抓一张高清 JPEG。
5. 如果 ISAPI 抓图成功，会在这张高清图上重新做人脸检测，再按检测框裁人脸。
6. 裁剪后再做质量过滤。

质量过滤规则在 `src/utils/quality.py` 里：

1. 宽度至少 `80` 像素。
2. 高宽比在 `0.6 ~ 1.6` 之间。
3. 拉普拉斯方差大于等于 `capture.quality_blur_th`，当前值是 `50`。

## 5.4 人脸库构建怎么做

文件：

- `src/recognize/gallery.py`

启动时会执行：

1. 扫描 `photo/` 目录。
2. 文件名去掉扩展名后作为身份名；如果有下划线，只取下划线前面的部分作为姓名。
3. 对每张图库照片先做人脸检测。
4. 取最大的那张脸，扩一圈后送入 ArcFace。
5. 同一人的多张图特征求平均，做 L2 归一化。
6. 保存到 `cache/gallery.npz`，下次如果源图片没变就直接加载缓存。

## 6. PTZ 控制逻辑

## 6.1 PTZ 控制分成两层

文件：

- `src/sdk/hik_sdk.py`
- `src/sdk/hik_ptz.py`
- `src/sdk/hik_isapi.py`

当前分工：

1. 海康 SDK：
   - 登录登出
   - 调预置位
   - 开始/停止录像
   - SDK 版 3D 定位接口 `NET_DVR_PTZSelZoomIn_EX`

2. ISAPI：
   - 读取高清 JPEG
   - 查询 PTZ 当前绝对位置
   - 发送绝对 PTZ 定位和绝对 zoom

## 6.2 当前实际使用的是哪条变倍链路

当前配置是：

```yaml
ptz:
  zoom_backend: isapi
```

所以 `ptz.zoom_to_bbox()` 走的是 **ISAPI absoluteEx** 路径，不是 SDK 的 `NET_DVR_PTZSelZoomIn_EX`。

当前 ISAPI 变倍逻辑：

1. 读取当前 PTZ 状态：
   - `azimuth`
   - `elevation`
   - `absoluteZoom`
2. 根据目标框中心相对画面中心的偏移，估算应补多少水平/垂直角度。
3. 根据目标框大小，估算应该把 zoom 拉到多少，目标是让目标大约占到画面的 `85%`。
4. 调 `ptz_absolute_zoom()` 把新的 `azimuth/elevation/absoluteZoom` 一次性发给相机。

也就是说，当前不是简单做“画框放大”，而是：

先读取球机当前位置，再反推出新的绝对 PTZ 目标位姿，然后把球机一次性打过去。

## 6.3 预置位跳转怎么做

调用的是：

- `HikPTZ.goto_preset()`
- 底层对应 `NET_DVR_PTZPreset_Other(..., GOTO_PRESET, preset_id)`

当前预置位相关行为：

1. 巡航阶段靠预置位跳转。
2. 扫描阶段每到一个预置位都会再次跳过去。
3. 每个目标识别完后，会回当前扫描预置位，再扫描剩余目标。

## 6.4 画面“稳定”是怎么判断的

文件：

- `src/scheduler/state_machine.py`

PTZ 指令发出后，状态机会做两段式稳定判定：

1. 先等最小等待时间：
   - 普通 PTZ 命令：`ptz.min_wait_after_cmd`
   - zoom 命令：`ptz.min_wait_after_zoom`
2. 等待结束后，主动 `flush()` 几帧 RTSP，避免读到陈旧画面。
3. 用相邻灰度帧的平均绝对差判断有没有运动。
4. 当画面差值连续若干帧低于 `settle_diff_th` 时，认为运动停止。

当前配置：

```yaml
ptz:
  min_wait_after_cmd: 0.8
  min_wait_after_zoom: 1.0
  motion_th: 15.0
  settle_diff_th: 5.0
  stable_frames: 3
```

## 6.5 对焦完成是怎么判断的

还是在 `state_machine.py` 里。

逻辑：

1. 对当前帧计算拉普拉斯方差。
2. 保留最近 3 帧清晰度值。
3. 只有当这 3 帧非递减，且最后一帧超过 `focus_min_laplacian`，才认为对焦完成。

当前阈值：

```yaml
ptz:
  focus_min_laplacian: 50.0
```

## 6.6 CAPTURE 阶段的动态修正逻辑

文件：

- `src/scheduler/capture_tracker.py`

设计上它支持：

1. 安全区内正常收集。
2. 脸偏出安全区时触发一次离散 PTZ 修正。
3. 短时丢脸时用 Kalman 预测脸中心。
4. 持续丢失时给出 `giveup`。

但是当前配置里：

```yaml
capture:
  tracking:
    max_corrections: 0
```

这会导致一个很关键的现状：

当前主流程里 CAPTURE 阶段的“修正”实际上是**关闭的**。  
只要脸落到安全区外，`CaptureTracker` 会直接给出 `giveup`，不会真的再打一轮 PTZ 修正。

所以现在 CAPTURE 阶段更接近：

1. 脸在安全区内就收集。
2. 脸偏得太厉害就放弃。
3. 依赖第一次 zoom 足够准，而不是依赖二次修正。

## 7. 识别逻辑

## 7.1 输入是什么

识别阶段输入不是整帧，而是 `capture_buf` 里累积的人脸 crop。

这些 crop 来自：

1. `face_close` 在近景里找到的人脸。
2. 优先在 ISAPI 高清抓图上重检后裁出的脸。
3. 通过 `quality_ok()` 质量过滤后的结果。

## 7.2 什么时候开始识别

在 `SCAN_CAPTURE` 中：

1. 收集到 `capture.min_samples` 张就立刻进入识别。
2. 或者超过 `capture.timeout`，也会结束采集并进入识别。

当前值：

```yaml
capture:
  min_samples: 3
  timeout: 3.0
```

## 7.3 识别判定怎么做

文件：

- `src/recognize/arcface.py`
- `src/recognize/gallery.py`
- `src/scheduler/state_machine.py`

流程：

1. 对 `capture_buf` 里的每张脸做 ArcFace embedding。
2. 多张 embedding 求平均。
3. 再做一次 L2 归一化。
4. 与人脸库矩阵做点积，相当于 cosine similarity。
5. 取 top1 相似度。

当前阈值：

```yaml
recognize:
  match_th: 0.35
  reject_th: 0.20
  min_samples_for_stranger: 3
```

当前判定规则：

1. `sim >= 0.35`，结果是 `hit`。
2. `sim < 0.20`，结果是 `stranger`。
3. 中间区间，结果是 `ambiguous`。

另外还有一个样本数保护：

如果最终抓到的有效脸图数量少于 `min_samples_for_stranger`，系统不会判成 `STRANGER`，而是记成 `UNRECOGNIZED`。

## 7.4 识别结果怎么落地

识别结束后：

1. 事件写入 `output/events.jsonl`。
2. 如果结果是 `stranger`，会把最后/最大的一张抓拍图保存到 `output/strangers/`。
3. 如果结果是 `hit`，会把对应目标加入本轮 `identified` 表，用于跨预置位去重。

一个重要细节：

当前代码只有 `hit` 会加入 `identified`。  
`stranger` 和 `ambiguous` 不会加入去重表。

这样做的效果是：

1. 已识别成功的人不会在别的预置位重复扫描。
2. 陌生人和模糊结果仍然可能在后续预置位再次被扫描。

## 8. ReID / 跟踪逻辑

文件：

- `src/track/sort_reid.py`

当前跟踪器的作用不是做长时 MOT，而是服务两个目的：

1. 在当前预置位扫描时给目标维持一个 track。
2. 在同一轮多预置位扫描里，用 ReID 做“已识别成功人员”的跳过。

具体实现：

1. 每个检测目标都会裁一个更大的区域。
2. 这个区域送进 `OSNetReID.embed()`，得到 512 维归一化特征。
3. `Tracker.update()` 用 `IoU + ReID` 的加权 cost 做匈牙利匹配。
4. 更新时对 `reid_feat` 做 EMA。
5. `ScanScheduler._is_already_identified()` 会拿当前 track 的 ReID 特征和 `identified` 表里的特征做点积比较。

当前配置：

```yaml
track:
  iou_weight: 0.6
  reid_weight: 0.4
  max_age: 30
  reid_ema: 0.1

reid:
  cross_preset_th: 0.5
```

## 10. 当前代码现状和需要特别注意的点

下面这些不是“设计想法”，而是当前代码真实现状：

### 10.1 巡航/扫描阶段主要检测的是人，不是脸

虽然项目里有 `face_wide`，但当前 `ScanScheduler` 在 `person_det` 存在时优先使用 `person_det`。  
因此远景阶段的目标框本质上是**人体框**，不是人脸框。

### 10.2 CAPTURE 阶段的二次修正当前等于关闭

因为 `capture.tracking.max_corrections = 0`，脸一旦不在安全区内，当前逻辑会直接放弃，不会继续做二次拉正。

### 10.3 YAML 里有一批参数目前没有真正接入主流程

当前代码里以下配置项并没有真正影响运行结果，或者没有被完整接线：

- `ptz.focus_wait_fixed`
- `ptz.zoom_min`
- `ptz.zoom_max`
- `ptz.hfov_1x`
- `ptz.vfov_1x`
- `ptz.settle_timeout`
- `track.min_hits`
- `capture.max_samples`
- `capture.tracking.enabled`

尤其要注意：

`ptz.hfov_1x` 和 `ptz.vfov_1x` 虽然写在 YAML 里，但当前 `HikPTZ` 没有从配置里接收这两个值。  
因此 ISAPI 变倍时实际还在使用 `hik_ptz.py` 里的默认值 `58` 和 `33`，不是 YAML 里写的 `100` 和 `67`。

### 10.4 有些代码存在，但当前状态机没有用到

目前存在但未接入主流程的内容包括：

- `Tracker.restore_after_move()`
- `EventLogger` 类
- `HikISAPI.ptz_drag_zoom()`
- `State.SCAN_ZOOM_IN` 基本只是过渡状态，实际从 `SCAN_PICK` 直接跳到 `SCAN_SETTLE`

### 10.5 `SCAN_ZOOM_OUT` 后没有单独再进一次“等待稳定”状态

当前实现里，识别完一个目标后会先 `goto_preset(current_scan_preset)`，然后直接把状态切回 `SCAN_PICK`，没有额外单独进一个回位后的检测稳定状态。  
这意味着“回预置位后马上挑下一个目标”是当前实现的一部分。

### 10.6 启动脚本里的 web 端口提示和配置不一致

`run.sh` 里打印的是：

```text
Web stream: http://0.0.0.0:18080/
```

但当前配置文件实际端口是：

```yaml
display:
  web_port: 18060
```

真正生效的是配置文件里的 `18060`。
