# Stage1 Baseline 说明

本文档记录分支 `20260416codex` 上第一阶段 `stage1_single_static` 的基线实现、实测结果和已知问题。

## 1. 目标定义

Stage1 的目标限定为：

- 默认全景
- 检测到单人后，驱动 PTZ 做人体 / 人脸居中
- 在满足窗口条件后持续 `zoom in`
- 最大 zoom 后保持 2 秒
- 保持结束后自动恢复到全景预置位
- 当目标丢失时也自动恢复到全景预置位

当前默认全景预置位：

- `home_preset = 1`

配置文件：

- `config/common.yaml`
- `config/stage1_single_static.yaml`

入口：

- `scripts/stage1_single_static.py`
- `src/main.py`

## 2. 当前实现

### 2.1 主流程

当前主状态机位于：

- `src/stage1_single_static.py`

当前状态包括：

- `SEARCH_PERSON`
- `CENTER_PERSON`
- `CENTER_FACE`
- `ZOOM_STEP`
- `HOLD_FACE`
- `RESTORE_HOME`
- `LOST_TARGET`

实际行为：

1. 程序启动后先执行 `restore_home("startup")`
2. 进入 `RESTORE_HOME`
3. 等待 `restore_home_settle_s`
4. 回到 `SEARCH_PERSON`
5. 检测到人后，根据人脸 / 人体位置做离散 PTZ 修正
6. 满足窗口条件后做 `zoom in`
7. `zoom_steps` 达到上限后，进入 `HOLD_FACE`
8. 保持 2 秒后执行 `restore_home("hold_timeout")`
9. 如果中途目标完全丢失，执行 `restore_home("lost_target")`

### 2.2 控制模式

当前 PTZ 控制不是连续伺服，而是离散脉冲控制：

- 发指令
- 保持固定 pulse 时间
- 发 stop
- 等待 settle

这一点是当前 Stage1 的明确设计选择，目的是优先跑通和稳定调试。

### 2.3 Web 模式

支持两种运行模式：

1. 普通模式
   - 启动 Web MJPEG 调试页
2. `--no-web`
   - 关闭 Web，减少调试开销，便于看时序日志

## 3. 时序日志

当前已经接入的日志包括：

- 周期摘要：
  - `[TIMING] summary`
- 指令发起：
  - `[TIMING] cmd=...`
- zoom 准备就绪：
  - `[TIMING] zoom_ready target=face ...`
- 稳定结束：
  - `[TIMING] settle_done reason=...`
- 进入保持：
  - `[TIMING] hold_face ...`
- 恢复全景：
  - `[TIMING] restore_home reason=...`
  - `[TIMING] restore_home_done home_preset=...`

这些日志已经足够支持后续 Stage2 调参和问题定位。

## 4. 实测结论

基于当前多轮实测日志，Stage1 基线可以认为已经成立。

### 4.1 已确认成立的行为

#### 启动回全景

日志链路已确认：

- `restore_home reason=startup`
- `STATE SEARCH_PERSON -> RESTORE_HOME`
- `restore_home_done home_preset=1`
- `STATE RESTORE_HOME -> SEARCH_PERSON`

说明：

程序启动后不会继承上一次残留的放大视角，而是先回到全景预置位。

#### 无人时保持全景

当无人时，状态会稳定停留在：

- `SEARCH_PERSON`

不会再长时间停在局部放大视角。

#### 有人时进入放大

日志中已经多次确认：

- `STATE ... -> ZOOM_STEP`
- `cmd=zoom_in`

说明：

当前实现已经能够进入实际的 zoom 链路，不再是“只转不放大”。

#### 最大 zoom 后进入保持

日志中已经确认：

- `STATE ZOOM_STEP -> HOLD_FACE`
- `hold_face total_ms=... zoom_steps=8`

说明：

当前 `max_zoom_steps=8` 的上限逻辑能走通。

#### 保持 2 秒后恢复全景

日志中已经确认：

- `restore_home reason=hold_timeout zoom_steps=8`
- `STATE HOLD_FACE -> RESTORE_HOME`
- `restore_home_done home_preset=1`

说明：

当前基线已满足“放大到最大后保持 2 秒并回全景”的核心要求。

#### 丢目标后恢复全景

日志中已经确认：

- `STATE ... -> LOST_TARGET (no_target)`
- `restore_home reason=lost_target`
- `STATE LOST_TARGET -> RESTORE_HOME`
- `restore_home_done home_preset=1`

说明：

当前已经修复“丢目标后停在局部视角不恢复”的问题。

## 5. 当前参数

当前 Stage1 关键参数在：

- `config/stage1_single_static.yaml`

重点参数：

- `home_preset: 1`
- `settle_after_move_s: 0.35`
- `settle_after_zoom_s: 0.80`

- `max_zoom_steps: 8`

## 6. 已知问题

### 6.1 轻微移动仍可能在进入 HOLD_FACE 之前打断流程

现象：

- 人有轻微移动、脸框抖动、构图变化时
- 在 `CENTER_FACE` / `CENTER_PERSON` / `ZOOM_STEP` 阶段仍可能被打断
- 从而提前走 `lost_target -> restore_home`

判断：

这已经更接近 Stage2 “单人移动追踪”的问题，不再作为 Stage1 blocker 处理。

### 6.2 PTZ 仍是离散脉冲式控制

现象：

- 控制是一步一步的，不是连续平滑追随

判断：

这是当前 Stage1 的接受范围，后续如需更顺滑追随，进入 Stage2 再处理。

### 6.3 检测耗时波动较大

从日志看，当前：

- `person_det_ms`
- `face_det_ms`
- `loop_ms`

波动仍然存在，尤其在脸检测开启后明显上升。

判断：

当前只要不破坏 Stage1 闭环即可接受；后续性能优化可作为独立议题。

## 7. 边界划分

### Stage1 结束标准

当前认为 Stage1 已达到基线结束标准：

- 启动回全景
- 无人时全景等待
- 有人时短时放大
- 能进入最大 zoom
- 保持 2 秒
- 自动恢复全景
- 丢目标也恢复全景

### Stage2 待解决问题

以下问题明确归入 Stage2：

- 单人轻微移动时的稳定追随
- 脸丢失时人体兜底的更稳切换
- 更强的目标粘连
- 更连续的 PTZ 控制策略
