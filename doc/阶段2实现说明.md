# Stage2 实现说明

当前版本对应 `stage2_single_moving` 单 PTZ 单人移动跟随实现。

目标：

- 单人移动时持续追随
- 人脸优先
- 丢脸时短时回退人体
- 目标彻底丢失后回到全景预置位
- 在无网页调试模式下保持较低控制延迟

## 当前入口

运行脚本：

- `scripts/stage2_single_moving.py`

核心代码：

- `src/stage2_single_moving.py`
- `src/ptz_controller.py`
- `src/detector.py`

配置文件：

- `config/common.yaml`
- `config/stage2_single_moving.yaml`

## 当前状态机

当前状态与代码一致：

- `SEARCH_WIDE`
- `ACQUIRE_TARGET`
- `ZOOM_TO_MAX`
- `FOLLOW_FACE`
- `FOLLOW_BODY`
- `LOST_TARGET`
- `RESTORE_HOME`

状态含义：

- `SEARCH_WIDE`
  - 预置位回中完成后的广角搜索阶段
  - 人脸优先进入 `ACQUIRE_TARGET`
  - 没有人脸但有人体时，以人体进入 `ACQUIRE_TARGET`
- `ACQUIRE_TARGET`
  - 使用预测后的目标中心做云台校正
  - 若已基本居中，则转入 `ZOOM_TO_MAX` 或直接进入 `FOLLOW_FACE`
- `ZOOM_TO_MAX`
  - 只在目标足够居中时触发 `zoom_in`
  - 人脸宽度达到阈值后直接转入 `FOLLOW_FACE`
- `FOLLOW_FACE`
  - 以人脸为主目标持续闭环跟随
  - 脸短暂丢失时允许切到 `FOLLOW_BODY`
- `FOLLOW_BODY`
  - 人脸丢失但人体仍在时的人体兜底跟随
  - 脸重新出现后切回 `FOLLOW_FACE`
- `LOST_TARGET`
  - 人脸和人体都不可用时的中间态
- `RESTORE_HOME`
  - 执行预置位回中并等待稳定

## 当前控制逻辑

1. 启动后先执行 `restore_home("startup")`
2. 等待 `restore_home_settle_s`
3. 进入 `SEARCH_WIDE`
4. 每帧同时运行：
   - `PersonDetector`
   - `FaceDetector`
5. 跟踪器使用单目标粘滞匹配：
   - 人体 tracker：`lost_timeout_s`
   - 人脸 tracker：`lost_face_to_body_s`
6. 目标误差由预测点计算：
   - 人脸使用 bbox 中心
   - 人体使用 `body_anchor_ratio` 对应的躯干锚点
   - 预测使用短时历史速度外推
7. 云台控制为非阻塞连续驱动：
   - 根据误差大小自适应计算速度和脉宽
   - `ptz.tick()` 每轮检查是否到时自动 stop
8. 放大策略：
   - 先满足居中条件
   - 若人脸宽度达到 `desired_face_ratio_min`，直接进入 `FOLLOW_FACE`
   - 否则执行 `zoom_in`
   - 最大放大步数受 `max_zoom_steps` 限制
9. 跟随超时：
   - 达到 `follow_duration_s` 后执行 `restore_home("follow_timeout")`
10. 完全丢目标：
   - 超过容忍时间后执行 `restore_home("lost_target")`

## 当前检测与执行链路

- 默认优先使用 GPU：
  - `runtime.prefer_gpu: true`
- 当前日志会打印 ONNX Runtime provider
- 实测日志中已确认：
  - `PersonDetector loaded ... providers=['CUDAExecutionProvider', 'CPUExecutionProvider']`
  - `FaceDetector loaded ... providers=['CUDAExecutionProvider', 'CPUExecutionProvider']`

当前 PTZ 控制器不是早期的阻塞式固定 pulse，而是支持：

- 异步 `pan_left_async / pan_right_async / tilt_up_async / tilt_down_async`
- 自动到时 stop
- 同方向指令续期
- 切换方向时先 stop 再发新指令

## 当前默认关键参数

来自 `config/stage2_single_moving.yaml`：

- `loop_sleep_s: 0.03`
- `settle_after_move_s: 0.08`
- `settle_after_zoom_s: 0.80`
- `restore_home_settle_s: 2.5`
- `follow_duration_s: 10.0`
- `lost_face_to_body_s: 0.5`
- `lost_all_restore_s: 1.0`
- `desired_face_ratio_min: 0.055`
- `max_zoom_steps: 6`
- `move_speed_min: 2`
- `move_speed: 5`
- `move_pulse_min_s: 0.05`
- `move_pulse_s: 0.18`
- `move_control_interval_s: 0.08`

这些参数对应当前思路：

- 先解决控制迟滞
- 放大策略偏保守
- 优先尽快进入 `FOLLOW_FACE`
- 不追求极限特写

## 当前日志特征

当前版本重点日志包括：

- `STATE ... -> ...`
- `[TIMING] summary ...`
- `[TIMING] cmd=... action=... decision_ms=...`
- `[TIMING] zoom_ready ...`
- `[TIMING] follow_ready ...`
- `[TIMING] follow_start ...`
- `[TIMING] restore_home ...`
- `[TIMING] settle_done ...`

其中：

- `action=pan_right@5:0.19s` 表示方向、速度、剩余驱动时长
- `decision_ms` 表示本轮从读帧到下发控制的决策耗时
- `follow_start` 表示已经进入正式跟随阶段

## 当前结论

截至当前代码和最近实测日志，Stage2 已形成可收敛基线：

- 检测推理已不再是主要瓶颈
- PTZ 跟随已从离散卡顿改为可持续闭环
- 已在实测中进入并保持 `FOLLOW_FACE`
- 跟随结束的主要退出原因已从“算不过来”转为配置控制逻辑，例如 `follow_timeout`

按当前目标，如果不是继续追求更极致的产品级丝滑，现版本可以作为 Stage2 baseline 暂时定档。

## 当前已知边界

当前版本仍存在这些已知边界，但不影响 baseline 定档：

- 从 `SEARCH_WIDE` 到稳定进入 `FOLLOW_FACE` 的时间仍有波动
- 二次重入稳定性不如首次进入
- 放大后视场变窄时，仍可能出现较快丢目标再回中的情况
- `follow_timeout` 目前是固定时长，不是无限持续跟随

后续若继续优化，应优先围绕：

- 缩短进入 `FOLLOW_FACE` 的时间
- 提高二次重入成功率
- 降低放大后的丢目标概率

不建议在当前阶段再回到 CPU 路线或阻塞 pulse 路线。
